# Design: 分层语义血缘图改造

## 架构边界

SQLGlot/SQLite 索引仍是血缘事实源，SAG 是结构化关系投影、事件证据检索和 Web 展示层。本变更只消费 SAG 现有 `LineageGraphRecord` 与按需图 API，不允许前端根据名称推断或创建血缘边。

GitNexus 与 codegraph 的索引生命周期、源码定位、影响分析等能力不在 UI 中伪造。对应差距和目标架构记录在 `docs/sql-codegraph-architecture-assessment.md`，后续由 SQL 索引内核独立演进。

## 画布模型

类型化血缘继续以 `LineageGraphRecord` 为数据源。画布模型把任务和表映射为顶层节点；字段通过 `table -> HAS_COLUMN -> column` 映射为表卡片内的字段行。没有所属表的字段保留为独立字段节点，避免丢失数据。

`HAS_COLUMN` 仅用于建立视觉归属，不绘制为普通关系线。其他关系保留原始事件 ID、证据数和上下文任务，并映射到以下端口：

- 任务或表端点连接节点左右主端口。
- 已显示字段端点连接对应字段行端口。
- 暂未显示的字段端点回退到所属表主端口。

表卡片默认只展示有限数量的有血缘字段，选中字段必须始终可见；用户可以在卡片内展开或收起其余字段。任何字段折叠只影响画布表达，不修改底层血缘图。

## 布局

使用 ELK layered 布局，方向固定为 `RIGHT`，关系使用正交路由。节点宽高由语义组件稳定计算后交给 ELK，布局结果再转换为 React Flow 坐标。字段折叠、搜索结果、穿透结果或类型筛选变化时重新布局并适配视图。

顶层节点数量继续受 500 个已加载实体上限约束；字段被收进表卡片后不再作为独立 DOM 节点。布局失败时必须保留可用的确定性分层回退布局并显示错误状态，不能返回空白画布。

## 工作台结构

桌面端采用三个协同区域：

- 左侧显示节点类型、关系类型、标签和穿透层级控制。
- 中央为白底 React Flow 分层画布，并保留搜索、复位、适配视图与缩放操作。
- 右侧为血缘检查器，显示当前实体、关系证据数量、当前穿透规模以及聚焦、展开和详情操作。

检查器没有选中项时显示当前图统计，不使用空白占位。窄视口优先保证中央画布，左右区域折叠为按需面板。

## 节点视觉

- 任务：紧凑加工节点，使用 Workflow 图标、任务名和关系计数。
- 表：白色矩形卡片，标题区显示数据库/模式、表名和字段统计，字段区显示有血缘字段行。
- 独立字段：紧凑字段卡片，用于没有 `HAS_COLUMN` 归属的数据。
- 选中节点使用高对比边框；关联节点保持类型色，无关节点降低透明度。

节点圆角不超过 6px，主信息必须在节点内直接可读，不依赖悬浮提示。长名称使用省略号并保留原文标题。

## 关系视觉

默认关系使用低干扰中性线、清晰目标箭头和正交路由。`LEFT JOIN`、`RIGHT JOIN`、`FULL OUTER JOIN`、`INNER JOIN` 等表级关系直接显示类型标签；其他关系在开启关系标签时显示，或仅在规模不超过 24 条的聚焦子图中自动显示，避免密集链路被重复标签覆盖。

选中实体时画布只布局当前穿透邻域节点和关系，避免已加载的大图把目标链路缩成不可读的缩略图；取消选择后恢复完整已加载图。点击关系继续打开对应事件详情。

## 兼容与性能

搜索、节点加载和多层穿透继续复用现有 API 与受限广度加载器。React Flow 自定义节点和边必须使用稳定类型注册，昂贵计算使用 memo；字段卡片不渲染无限列表。

移动端保留平移、缩放、搜索、层级控制和详情面板，筛选面板及节点详情不得超出视口。

## 验证

- 单元测试覆盖字段归属、字段折叠、端口映射、关系标签和无归属字段回退。
- 布局测试覆盖从左到右的层级方向、动态节点尺寸和空图。
- Playwright 在真实 SQL 血缘项目上验证白色非空画布、语义节点、箭头、字段展开、`1-2` 层穿透、空白恢复以及桌面/移动端边界。
- 浏览器验收同时检查左右工作区不遮挡中央画布，节点文本不溢出且关系标签可辨认。
